## 二、訂閱系統時程模型變更（前端必讀）

新 API 欄位（`chargeDate` / `t7Date` / `t8Date` / `nextChargeDate` / `nextShippingDate`）對應一套全新的時程模型。完整需求背景見 [`plans/出貨日.md`](../plans/出貨日.md)，這裡只給前端理解 API 所需的關鍵差異。

### 核心精神

| 維度 | 舊（2026-04 之前） | 新（2026-05 之後） |
|------|---------------------|---------------------|
| 推算原點 | 訂閱基準日（每月固定日） | 扣款日 T（每期實際扣款） |
| 出貨日 | 訂閱基準日起的第一個週三 | T 起的下一個週二（週一盤點、週二出貨） |
| 選香對象 | 本期即將出貨的香氣 | 下期香氣（本期自動沿用上期） |
| 鎖定窗口 | 出貨日前 3 天 | T 到 T+7（首期例外不鎖） |
| 預設行為 | 顧客本期未選 → 不出貨 | 顧客本期未選 → **沿用上期自動出貨** |

### 新時程節點（單期視角）

```
        T              T+7         T+8
        │   出貨日      │           │
        ├───┬──────────┼───────────┤────────── 下期 T ────────►
        │   │          │           │
       扣款 │   出貨     │           │
            │ (週二)    │           │
            │           │           │
            │  鎖定中    │  選香開放  │
            │ (不可改)   │  (可改下期) │
            ▼           │           │
       上期香氣          │ T+7 對顧客 │ 顧客可寫入
       自動沿用          │ 的出貨承諾 │ next_scent_name
       為本期            │ 截止日     │ 無限次直到下期 T
```

對應到 API 欄位：

| 時間點 | API 欄位 | 用途 |
|--------|---------|------|
| 本期 T | `chargeDate` | 顯示「本期扣款日」 |
| 本期出貨日 | `shippingDate` | 顯示「預計 X 出貨」 |
| T+7 | `t7Date` | 顯示「對顧客承諾的最晚出貨日」 |
| T+8 | `t8Date` | 顯示「下期選香開放日」；鎖定文案中也是此值 |
| 下期 T | `nextChargeDate` | 顯示「下次扣款日」（取代舊版 `deadline`） |
| 下期出貨日 | `nextShippingDate` | 顯示「下期預計出貨日」 |
| 當下狀態 | `isLocked` / `isSelectionOpen` | 切換選香按鈕是否可點 |

### 顧客體驗變化

**舊**：顧客在「截止日」前必須選香，否則影響本期出貨。
**新**：顧客**永遠不會錯過出貨** — 沒選的話自動沿用上期；選香變成「想換的話才動作」，且選的是下期。

對前端文案的影響：
- 「請於 X 前選香」→ 不再需要倒數提醒
- 「本期香氣未選」→ 改為「本期沿用上期：{lastPerfume}」
- 「截止日」→ 改為「下次扣款日」（`nextChargeDate`），語意是「在此之前都還可以修改下期」

### 期數概念（`currentPeriodNo`）

新版引入「期」的概念：
- 首期 = 1（會員剛綁定那期）
- 每次扣款後 +1
- 期數對齊曆月，但不一定精準（月底訂閱者可能有 ±1 期誤差，由 webhook 接入後校正）

`currentPeriodNo === 1` 時為「首期會員」，API 行為有兩個特例：
1. `isLocked` 永遠為 `false`（讓首期可隨時改香氣）
2. `select_perfume` 寫入 `current_scent_name` 而非 `next_scent_name`，回應 `appliedTo: "current"`

### 範例：實際會員當下狀態解讀

下方為下一節 cloud 實際回應的詮釋：

```
currentPeriodNo: 4         → 第 4 期（首期 = 2026-02 訂閱）
chargeDate:    2026-05-08  → 本期 5/8 扣款
shippingDate:  2026-05-12  → 5/12 (週二) 出貨
t7Date:        2026-05-15  → 對顧客承諾「5/15 前出貨」(已提前達成)
t8Date:        2026-05-16  → 5/16 起可改下期香氣 (= 今天)
nextChargeDate: 2026-06-08 → 下次扣款 6/8
nextShippingDate: 2026-06-09 → 下期出貨 6/9 (週二)
isLocked:      false        → 目前可選下期香氣
nextPerfume:   "沿用本期"   → 顧客還沒選下期，將自動用 currentPerfume
```

### 欄位對照（與 2026-04 版相比）

| 狀態 | 欄位 | 說明 |

## 六、`bind`（已部署、實際 cloud 錯誤路徑）

請求 / 成功回應與舊版一致；錯誤訊息也不變。下面是實際 curl 拿到的錯誤路徑回應：

### 訂單不存在（HTTP 404）

```bash
curl -X POST "https://kvqkjtyhbbcivwlexgcn.supabase.co/functions/v1/member" \
  -H "Content-Type: application/json" \
  -d '{"action":"bind","orderId":"NO_SUCH_ORDER_999","lineUid":"U_NEW_FAKE"}'
```

```json
{ "status": 404, "message": "找不到對應訂單編號" }
```

### 缺參數（HTTP 400）

```bash
curl -X POST "https://kvqkjtyhbbcivwlexgcn.supabase.co/functions/v1/member" \
  -H "Content-Type: application/json" \
  -d '{"action":"bind","orderId":"X"}'
```

```json
{ "error": "缺少必要參數 (orderId 或 lineUid)" }
```

### 重複綁定（HTTP 409） — 概念

實際情境需要傳入「已綁定者持有的真實 order_id」才會撞到此檢查；行為與舊版一致，沿用：
- `409` `"此 LINE 帳號已綁定訂閱"`
- `409` `"此訂單已被其他帳號綁定"`

> 後端內部：`members` 表已瘦身，`bind` 改為「UPSERT members 身份 + INSERT subscriptions（首期）+ INSERT charges（period_no=1）」。對前端來說只看回應，行為不變。

## 七、`select_perfume`（已部署、實際 cloud 錯誤路徑）

### 請求變更

| 參數 | 舊 | 新 |
|------|----|----|
| `action` | `"select_perfume"` | 不變 |
| `lineUid` | 必填 | 不變 |
| `scentName` | 必填 | 不變 |
| `displayName` | 選填 | **移除**（傳了會被忽略；會員顯示名由 `members` 表維護） |

### Request 範例

```bash
curl -X POST "https://kvqkjtyhbbcivwlexgcn.supabase.co/functions/v1/member" \
  -H "Content-Type: application/json" \
  -d '{"action":"select_perfume","lineUid":"<LINE_UID>","scentName":"柑橘花園"}'
```

### Response 200（成功）

舊版：
```json
{ "status": 200, "message": "選香紀錄成功！", "targetMonth": "2026-05", "scentName": "薰衣草" }
```

新版：
```json
{
  "status": 200,
  "message": "選香紀錄成功！",
  "currentPeriodNo": 2,
  "appliedTo": "next",
  "scentName": "柑橘花園"
}
```

| `appliedTo` | 含意 | 前端建議文案 |
|-------------|------|--------------|
| `"current"` | 首期會員（`currentPeriodNo === 1`），直接改本期香氣（出貨尚未發生） | 「本期選香：{scentName}」 |
| `"next"` | 一般情境，寫入下期；下次 T 扣款時 promote 為本期 | 「下期選香：{scentName}」 |

### 未綁定（HTTP 403） — 實際 cloud 回應

```bash
curl -X POST "https://kvqkjtyhbbcivwlexgcn.supabase.co/functions/v1/member" \
  -H "Content-Type: application/json" \
  -d '{"action":"select_perfume","lineUid":"U_NEW_FAKE","scentName":"白茶微風"}'
```

```json
{ "status": 403, "message": "找不到您的有效訂閱紀錄，請先完成綁定！" }
```

### 鎖定期（HTTP 403） — 訊息已更新

```json
{
  "status": 403,
  "message": "本期出貨準備中，下期選香將於 2026-05-22 開放。"
}
```

> 訊息中的日期會自動帶當期的 `t8Date`。前端若已自行處理鎖定文案，可改讀 `check_bound` 的 `t8Date`，不再需要 parse 錯誤訊息。

### 缺參數（HTTP 400） — 實際 cloud 回應

```json
{ "error": "缺少必要參數 (lineUid 或 scentName)" }
```

## 九、UI 切換建議

| 場景 | `isLocked` | 顯示重點 |
|------|-----------|----------|
| T 到 T+7（出貨準備中） | `true` | 「本期出貨準備中，下期選香將於 {t8Date} 開放」、隱藏選香按鈕 |
| T+8 到下期 T | `false` | 顯示「為下期挑選香氣」；若 `nextPerfume === "沿用本期"` 加上「不選將沿用本期」提示 |
| 首期會員（`currentPeriodNo === 1`） | `false` | 顯示「修改本期香氣」（API 會自動寫入本期，回應 `appliedTo: "current"`） |

範例顯示文字：

```
本期香氣：{currentPerfume}（{chargeDate} 扣款，預計 {shippingDate} 出貨）
下期香氣：{nextPerfume}
下次扣款：{nextChargeDate}
```

當 `isLocked === true`：
```
本期出貨準備中…
下期選香將於 {t8Date} 開放
```
